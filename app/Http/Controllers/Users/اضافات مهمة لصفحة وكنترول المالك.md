# ترتيب التنفيذ المقترح

## حالة التنفيذ الحالية

تم تنفيذ المراحل الأولى حتى السادسة، وتشمل:

- تعقيم اسم المتجر داخل `innerHTML`.
- استخدام مصدر المديونيات الفعلي وقيمة `remaining_amount` في المخطط.
- تجميع مؤشرات المتاجر حسب `store_id` بدل الاستعلام داخل الحلقة.
- حذف استعلام `storePaidToday` واستخدام مبيعات المتجر المحصلة مباشرة.
- إضافة Cache قصير لملخص الشهر وأفضل المنتجات والملخص اليومي الحي.
- تخزين نتيجة `Schema::hasColumn()` داخل متغير ثابت أثناء الطلب.
- إنشاء حساب تكلفة منتجات مجمع حسب المتجر.
- جعل التحديث كل خمس ثوانٍ وإيقافه عند إخفاء التبويب وإظهار حالة فشل الاتصال.
- تقسيم `index()` إلى دوال خاصة داخل الكنترولر نفسه.
- إنشاء Scope موحد للمبيعات المحتسبة.
- تجميع رواتب المتاجر باستعلام واحد خارج حلقة المتاجر.

لم يُضف WebSocket أو Laravel Reverb، ولم تتم إضافة فهارس أو migrations أو أي تعديل في قاعدة البيانات.

## المرحلة الأولى: تحسين أمني مباشر

### 1. تعقيم اسم المتجر قبل إدراجه داخل `innerHTML`

**الحالة: تم التنفيذ.**

نافذة تفاصيل البطاقات تبني صفوف المتاجر باستخدام `innerHTML`. لذلك يجب عدم وضع اسم المتجر داخل النص مباشرة، بل تعقيمه أولًا لمنع تفسير أي محارف داخله كوسوم HTML أو JavaScript.

تمت إضافة دالة تعقيم خاصة بنافذة المؤشرات:

```js
function escapeMetricHtml(value) {
    return String(value ?? '')
        .replaceAll('&', '&amp;')
        .replaceAll('<', '&lt;')
        .replaceAll('>', '&gt;')
        .replaceAll('"', '&quot;')
        .replaceAll("'", '&#039;');
}
```

ثم أصبح اسم المتجر يعرض بهذه الطريقة:

```js
escapeMetricHtml(store.store_name)
```

هذا التعديل أمني فقط ولا يغير شكل النافذة أو بياناتها أو طريقة عملها.

## المرحلة الثانية: تصحيح مصدر بيانات المديونيات

### 2. الاعتماد على الدين المتبقي بدل المبلغ المحصل من الآجل

المخطط يجب أن يعرض **الديون المتبقية فعليًا**، وليس `paid_amount` لعمليات البيع الآجل.

عند التنفيذ يجب الاعتماد على أحد المصدرين التاليين، حسب المصدر المعتمد في النظام:

1. `remaining_amount` من سجلات المبيعات إذا كان يمثل الدين الحالي بصورة موثوقة.
2. جدول المديونيات الفعلي مثل `employee_credit_sales` إذا كان هو المصدر الرسمي لحالة الدين والسداد.

يجب أن تكون القيمة المعروضة:

```text
إجمالي المبالغ المتبقية غير المسددة في اليوم أو الفترة
```

ويجب التحقق من:

- عدم إدخال المديونيات المسددة.
- عدم تكرار الدين إذا كان مرتبطًا بعملية بيع وسجل مديونية في الوقت نفسه.
- استخدام تاريخ المديونية المناسب للمخطط.
- تطابق الرقم مع صفحة المديونيات.

# المرحلة الثالثة: تحسين استعلامات الكنترولر

## 3. إجراء استعلام تجميعي واحد لكل نوع بيانات

بدل تنفيذ استعلامات مستقلة داخل حلقة كل متجر، تُجمع البيانات حسب `store_id`.

الأنواع التي يجب تجميعها:

- مبيعات اليوم.
- مبيعات الشهر.
- مصروفات اليوم.
- مصروفات الشهر.
- الرواتب.
- مشتريات المالك.
- الاستهلاك الداخلي.
- تكلفة المنتجات اليومية.
- تكلفة المنتجات الشهرية.

مثال:

```php
$salesTodayByStore = Sale::query()
    ->whereIn('store_id', $storeIds)
    ->whereDate('created_at', today())
    ->groupBy('store_id')
    ->selectRaw('store_id, COALESCE(SUM(paid_amount), 0) as total')
    ->pluck('total', 'store_id');
```

ثم تستخدم النتيجة داخل تجهيز تفاصيل المتاجر:

```php
$storeSalesToday = (float) ($salesTodayByStore[$storeId] ?? 0);
```

### الهدف

- تقليل عدد الاستعلامات.
- منع زيادة الاستعلامات مع زيادة المتاجر.
- جعل إجمالي الصفحة وتفاصيل المتاجر مبنيين على المصدر نفسه.

---

## 4. استخدام `storeSalesToday` مباشرة في حساب الربح

لا حاجة لتنفيذ استعلام آخر باسم `storePaidToday` إذا كانت قيمة `storeSalesToday` تحتوي بالفعل على مجموع `paid_amount` بالشروط نفسها.

يصبح الحساب:

```php
$storeProfitToday =
    (float) $storeSalesToday
    - (float) $storeProductsCostToday;
```

وعند اعتماد خصم المصروفات من الربح اليومي مستقبلًا يجب تطبيق المعادلة نفسها في الصفحة والتحديث الحي:

```php
$storeProfitToday =
    (float) $storeSalesToday
    - (float) $storeProductsCostToday
    - (float) $storeExpensesToday;
```

لا تُغير معادلة الربح قبل اعتماد تعريف الربح المطلوب بصورة نهائية.

---

## 5. تحسين جلب أفضل خمسة منتجات باستخدام Cache قصير

المرحلة الأولى تكون بتخزين نتيجة أفضل خمسة منتجات لمدة قصيرة، مثل خمس دقائق:

```php
Cache::remember(
    "owner-dashboard:{$user->id}:top-products:".now()->format('Y-m'),
    now()->addMinutes(5),
    fn () => /* استعلام أفضل المنتجات */
);
```

بعد نمو البيانات يمكن تحسين استعلام SQL نفسه باستخدام ترتيب داخل قاعدة البيانات بدل جلب جميع النتائج ثم قصها في PHP.

### ملاحظات

- يجب أن يكون مفتاح Cache خاصًا بالمالك والشهر.
- يجب ألا تختلط نتائج مالك بمالك آخر.
- مدة خمس دقائق مناسبة لأن هذه إحصائية شهرية وليست بطاقة لحظية.

# المرحلة الرابعة: تحسين حساب تكلفة المنتجات

## 6. تخزين نتيجة فحص عمود التكلفة داخل متغير ثابت

بدل تنفيذ `Schema::hasColumn()` عند كل استدعاء للدالة خلال الطلب نفسه:

```php
static $hasStoredItemCosts;

$hasStoredItemCosts ??= Schema::hasColumn(
    'sale_items',
    'total_cost'
);
```

ثم:

```php
if (! $hasStoredItemCosts) {
    // الحساب التوافقي للعمليات القديمة.
}
```

هذا التخزين يعمل أثناء الطلب الحالي فقط ولا يغير قاعدة البيانات.

---

## 7. إنشاء دالة تعيد التكلفة مجمعة حسب المتجر

بدل استدعاء دالة تكلفة المنتجات مرة لكل متجر، تنشأ دالة داخل `UserDashboardController` تعيد مصفوفة مثل:

```php
[
    1 => 150.00,
    2 => 320.50,
    3 => 0.00,
]
```

توقيع مقترح:

```php
private function calculateProductsCostByStore(
    $storeIds,
    $start,
    $end,
    array $saleTypes
): array
```

يجب أن:

- تستخدم `sale_items.total_cost` عند اكتمال تكاليف أسطر العملية.
- تستخدم الحساب التوافقي للعملية القديمة عند نقص تكاليف الأسطر.
- تعتبر عملية شغل اليد دون منتجات ذات تكلفة منتجات تساوي صفرًا.
- تعيد صفرًا للمتجر الذي لا توجد له عمليات.

---

## 8. تخزين ملخص الشهر مؤقتًا

يمكن تخزين البيانات الشهرية الثقيلة لعدة دقائق، مثل:

- مبيعات الشهر.
- تكلفة منتجات الشهر.
- مصروفات الشهر.
- مشتريات المالك.
- الاستهلاك الداخلي.
- أفضل المنتجات.

يجب أن يكون مفتاح Cache:

- خاصًا بالمالك.
- خاصًا بالشهر والسنة.
- مختلفًا عند وجود فلتر يؤثر على النتيجة.

مدة مقترحة:

```text
من 3 إلى 5 دقائق
```

أما البيانات اليومية الحية فلا تستخدم Cache شهريًا طويلًا.

---

## 9. عدم حساب بيانات الشهر داخل التحديث الحي

مسار `dailySnapshot` يجب أن يبقى مسؤولًا عن بيانات اليوم فقط:

- مبيعات اليوم.
- مصروفات اليوم.
- تكلفة اليوم.
- الربح اليومي.
- عدد العمليات.
- آخر عملية.

لا ينبغي أن يعيد حساب:

- ملخص الشهر.
- الرواتب.
- أفضل المنتجات.
- المخزون المنخفض.
- مخطط 14 يومًا.

بهذا يبقى التحديث الحي خفيفًا.

## 10. الفهارس المقترحة

قبل إضافة أي فهرس يجب مراجعة الفهارس الموجودة بالفعل باستخدام أدوات قاعدة البيانات.

الحقول المرشحة للمراجعة:

- `sales.store_id`
- `sales.created_at`
- `sales.sale_type`
- `sale_items.sale_id`
- `sale_items.product_id`
- `expenses.store_id`
- `expenses.created_at`
- حقول مصدر المديونيات المستخدمة في الحالة والتاريخ والمتجر

> **مهم:** لا يضاف أي فهرس، ولا ينشأ migration، ولا تعدل قاعدة البيانات إلا بعد موافقة صريحة.

# المرحلة الخامسة: تحسين التحديث الحي

## 11. تغيير فترة التحديث

تتغير الفترة من ثلاث ثوانٍ إلى خمس أو عشر ثوانٍ.

التوصية:

```text
5 ثوانٍ
```

وهي فترة تحافظ على الإحساس بالتحديث السريع وتخفض عدد الطلبات.

---

## 12. إيقاف التحديث عندما يكون التبويب مخفيًا

قبل إرسال الطلب:

```js
if (document.hidden) {
    return;
}
```

ويجب الاستماع إلى `visibilitychange` لاستئناف التحديث فور عودة المستخدم:

```js
document.addEventListener('visibilitychange', function () {
    if (!document.hidden) {
        refreshDailySnapshot();
    }
});
```

هذا لا يغير التحديث التلقائي الموجود، بل يمنع الطلبات غير المفيدة عندما لا تكون الصفحة ظاهرة.

---

## 13. الاحتفاظ بآخر معرف عملية

الواجهة تحتفظ بالفعل بآخر معرف عملية لتمييز وصول عملية جديدة.

يمكن توسيع ذلك بحيث:

- يرسل المتصفح آخر معرف يعرفه.
- يعيد الخادم تفاصيل آخر عملية فقط إذا تغير المعرف.
- تبقى الأرقام الإجمالية محدثة في كل طلب.

هذا تحسين لاحق، ولا يجب أن يعطل بطاقة آخر عملية الحالية.

---

## 14. Cache قصير جدًا للملخص اليومي

يمكن تخزين الحسابات الثقيلة لمدة ثانيتين أو ثلاث ثوانٍ لمنع إعادة الحساب نفسه عندما تكون الصفحة مفتوحة في عدة تبويبات.

يجب ألا يمنع ذلك ظهور العملية الجديدة بعد انتهاء المدة القصيرة.

مفتاح Cache يجب أن يتضمن:

- معرف المالك.
- معرف المتجر المحدد أو عبارة `all`.
- تاريخ اليوم.

---

## 15. WebSocket أو Laravel Reverb

يُدرس لاحقًا فقط إذا احتاج النظام إلى تحديث فوري حقيقي دون polling.

حاليًا يكفي:

- تحديث كل خمس ثوانٍ.
- إيقاف التحديث في التبويب المخفي.
- Cache قصير جدًا.

لا حاجة لإضافة Reverb في المرحلة الحالية.

# المرحلة السادسة: تنظيم الكنترولر دون إنشاء ملفات جديدة

## 16. تقسيم دالة `index()` داخل الكنترولر نفسه

يجب تقسيم الدالة إلى دوال خاصة داخل `UserDashboardController` فقط، دون إنشاء Service أو ملفات إضافية.

الدوال المقترحة:

```php
private function resolveDashboardStores($user): array
private function buildDailySummary($storeIds, array $saleTypes): array
private function buildMonthlySummary($user, $storeIds, array $saleTypes): array
private function buildSalarySummary($user, $storeIds): array
private function buildCreditSummary($storeIds): array
private function getLowStockSummary($storeIds): array
private function getTopSellingProducts($user, $storeIds, array $saleTypes)
private function buildStoreBreakdowns($stores, array $saleTypes): array
private function buildLatestOperation($salesQuery): ?array
```

تبقى دالة `index()` مسؤولة عن:

1. جلب المستخدم.
2. تحديد المتاجر.
3. استدعاء الدوال الخاصة.
4. دمج البيانات.
5. إرسالها إلى الواجهة.

### شروط التقسيم

- عدم تغيير أسماء المتغيرات التي تعتمد عليها الواجهة إلا عند تحديثها بصورة آمنة.
- عدم تغيير المعادلات أثناء إعادة التنظيم.
- عدم إنشاء ملفات جديدة.
- إبقاء `dailySnapshot` داخل الكنترولر نفسه.
- تشغيل اختبارات مقارنة قبل وبعد التقسيم.

## 17. إنشاء Scope موحد للمبيعات المحتسبة

ينشأ Scope داخل نموذج `Sale` لتوحيد:

- أنواع المبيعات المحتسبة.
- استبعاد الفاتورة اليدوية.

مثال:

```php
public function scopeCollectedDashboardSales($query)
{
    return $query
        ->whereIn('sale_type', ['cash', 'card', 'credit', 'mixed'])
        ->where(function ($query) {
            $query->whereNull('description')
                ->orWhere('description', '!=', 'manual_invoice_entry');
        });
}
```

ثم يستخدم في:

- `index()`.
- `dailySnapshot()`.
- تفاصيل المتاجر.
- استعلام أفضل المنتجات عندما يكون ذلك مناسبًا.

### الهدف

- منع اختلاف شروط المبيعات بين البطاقة والتحديث الحي.
- تقليل التكرار.
- تسهيل تعديل الشرط في مكان واحد.

# المرحلة السابعة: تحسين الواجهة دون تغيير النظام الحالي

## 18. فلتر المتجر للملخص اليومي

يضاف فلتر صغير أعلى الملخص اليومي:

```text
جميع المتاجر
متجر 1
متجر 2
...
```

### الشروط

- عدم إزالة التحديث التلقائي.
- عند اختيار متجر يستمر `dailySnapshot` في العمل.
- يرسل `summary_store_id` مع كل طلب تحديث.
- عند اختيار «جميع المتاجر» يزال الفلتر.
- تتحدث البطاقات وعدد العمليات وآخر عملية بحسب الاختيار.
- لا يسمح الخادم بعرض متجر لا يتبع المالك.
- لا تتغير روابط بطاقات المتاجر الحالية.

يمكن حفظ الاختيار في عنوان الصفحة أو `localStorage`، لكن المصدر النهائي للصلاحية يبقى الخادم.

## 19. تحسين المخطط

التحسينات المطلوبة:

- استخدام مصدر الدين المتبقي الحقيقي.
- توضيح أن المبيعات هي المبالغ المحصلة.
- إضافة Tooltip يعرض التاريخ والقيمة.
- إضافة اختيارات 7 و14 و30 يومًا.
- الحفاظ على التصميم الحالي والألوان ما أمكن.
- عدم تحميل بيانات المخطط ضمن `dailySnapshot`.
- جلب بيانات المخطط فقط عند تحميل الصفحة أو عند تغيير الفترة.

## 20. توحيد تنسيق العملة

يجب اعتماد تنسيق موحد في:

- البطاقات.
- نافذة تفاصيل البطاقات.
- نافذة الرواتب.
- آخر عملية.
- قائمة أفضل المنتجات.

صيغة مقترحة:

```text
1,250.00 ر.س
```

ويفضل إنشاء دالة JavaScript واحدة:

```js
function formatCurrency(value) {
    return `${Number(value || 0).toLocaleString('en-US', {
        minimumFractionDigits: 2,
        maximumFractionDigits: 2,
    })} ر.س`;
}
```

وفي Blade يستخدم `number_format($value, 2)` بصورة موحدة.

يجب تجنب:

- عرض العملة مرتين.
- عرض بعض القيم دون كسور وأخرى بكسور دون سبب.
- اختلاف صيغة البطاقة عن نافذة التفاصيل.

---

## 21. مزامنة نافذة التفاصيل مع القيم الحية

القيم الموجودة في `metricDefinitions` يجب تحديثها بعد كل استجابة ناجحة من `dailySnapshot`.

مثال:

```js
metricDefinitions.profit_today.value = formatCurrency(data.profit_today);
metricDefinitions.sales_today.value = formatCurrency(data.sales_today);
metricDefinitions.expenses_today.value = formatCurrency(data.expenses_today);
metricDefinitions.products_cost_today.value = formatCurrency(data.products_cost_today);
```

أما تفاصيل المتاجر فيفضل جلبها عند فتح النافذة بدل إرسالها كل خمس ثوانٍ.

بهذا:

- تبقى البطاقة محدثة.
- تعرض النافذة الرقم نفسه.
- لا يصبح طلب التحديث الحي ثقيلًا.

---

## 22. إظهار حالة فشل الاتصال

يضاف عداد لعدد محاولات التحديث الفاشلة:

```js
let consecutiveSnapshotFailures = 0;
```

عند النجاح:

```js
consecutiveSnapshotFailures = 0;
```

عند الفشل:

```js
consecutiveSnapshotFailures++;
```

بعد فشل طلبين أو ثلاثة:

- تتحول نقطة الحالة إلى اللون البرتقالي.
- يظهر نص «تعذر التحديث».
- يبقى آخر وقت تحديث ناجح ظاهرًا.

عند نجاح الطلب التالي:

- تعود النقطة إلى الأخضر.
- يظهر وقت التحديث الجديد.

لا يجب إخفاء القيم الحالية عند فشل الاتصال؛ تبقى آخر قيم ناجحة ظاهرة.

# ترتيب التنفيذ النهائي

## أولوية أولى

1. تعقيم اسم المتجر داخل `innerHTML` — **تم التنفيذ**.
2. حذف استعلام `storePaidToday` المكرر.
3. تخزين نتيجة `Schema::hasColumn()` أثناء الطلب.
4. تغيير التحديث إلى خمس ثوانٍ وإيقافه عند إخفاء الصفحة.
5. إظهار حالة فشل الاتصال.
6. مزامنة نافذة التفاصيل مع القيم الحية.

## أولوية ثانية

7. تقسيم `index()` داخل الكنترولر نفسه.
8. إنشاء Scope موحد للمبيعات.
9. تجميع الاستعلامات حسب `store_id`.
10. إنشاء حساب تكلفة مجمع حسب المتجر.
11. إضافة Cache قصير للبيانات الشهرية وأفضل المنتجات.

## أولوية ثالثة

12. إضافة فلتر المتجر مع الحفاظ على التحديث التلقائي.
13. تصحيح مصدر المديونيات في المخطط.
14. تحسين المخطط والفترات والـ Tooltip.
15. توحيد تنسيق العملة.
16. مراجعة الفهارس بعد الحصول على موافقة صريحة.

# ضوابط التنفيذ

- لا تعدل قاعدة البيانات دون موافقة صريحة.
- لا تضف migration أو فهرسًا في هذه المراحل دون موافقة.
- لا تغير معادلات الربح ضمن إعادة التنظيم.
- لا تلغِ التحديث التلقائي.
- لا تنشئ ملفات تنظيمية جديدة عند تقسيم `index()`؛ التقسيم يكون داخل الكنترولر.
- يجب أن تبقى بيانات كل مالك محصورة في متاجره.
- يجب مقارنة أرقام الصفحة قبل وبعد كل تحسين.
- يفضل تنفيذ كل مرحلة في commit مستقل لتسهيل المراجعة والتراجع.
