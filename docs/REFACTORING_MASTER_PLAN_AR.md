# المستند الموحد بعد اكتمال إعادة التنظيم المالي والموظفين و SRP

> **آخر تحديث:** 2026-07-01
> **الحالة:** مكتمل برمجيًا ضمن نطاق المستندات الثلاثة السابقة، بانتظار التجربة اليدوية والمقارنة.
> **بديل عن:** `FINANCIAL_REFACTORING_ROADMAP_AR.md`, `EMPLOYEE_ACCOUNTANT_OPERATIONS_REVIEW_AR.md`, `SINGLE_RESPONSIBILITY_SERVICE_EXTRACTION_AR.md`.

---

## 1. القرار الحالي

تم دمج المستندات الثلاثة في هذا المستند حتى لا تبقى تعليمات قديمة متعارضة مع الحالة الحالية.

القرار الحالي:

1. لا يوجد تنفيذ برمجي جديد مطلوب قبل التجربة اليدوية بعد تنظيف صفحة عمليات الموظف/المحاسب.
2. لا يتم حذف أو تصفير بيانات مالية أو عمليات قديمة تلقائيًا.
3. أي تنظيف لاحق للحسابات أو fallback القديم يجب أن يبدأ بتقرير تدقيق ثم حذف آمن للكود الزائد فقط.
4. صلاحيات المحاسب لا تتغير في هذه المرحلة.
5. التجربة اليدوية تبدأ قبل أي refactor مالي كبير جديد.

---

## 2. ما اكتمل برمجيًا

### 2.1 الملخص المالي والتكلفة

تم تنفيذ مصدر مالي مركزي عبر:

- `App\Data\Finance\StoreFinancialSummary`
- `App\Data\Finance\FinancialSummaryResult`
- `App\Services\Accounting\FinancialSummaryService`
- `App\Services\Accounting\SalesCostService`

القاعدة الحالية:

- الملخصات المالية يجب أن تأتي من الخدمة المركزية.
- تكلفة المبيعات الجديدة تعتمد على `sale_items.total_cost`.
- أي fallback قديم يبقى مؤقتًا فقط لحماية البيانات التاريخية إلى حين انتهاء المقارنة.

### 2.2 الشفت والتاريخ المحاسبي

تم تثبيت القواعد التالية:

- `created_at` يبقى وقت الإدخال الحقيقي.
- `business_date` هو التاريخ المحاسبي للتقارير.
- `daily_balance_id` يربط العملية بالشفت عند توفره.
- `ShiftOperationBinderService` هو نقطة ربط/نقل العمليات الأساسية مع الشفت.
- `HasAccountingDateScopes` هو نقطة fallback المحاسبي الموحدة للبيانات القديمة.

### 2.3 عمليات الموظفين والمحاسبين

تم توحيد عمليات الموظف الأساسية عبر:

- `EmployeeOperationService`
- `EmployeeHistoricalStoreService`
- `EmployeePayrollService`
- `EmployeeActionsViewService`

القواعد الحالية:

- المحاسب حساب دخول، وعملياته كعامل تسجل على الموظف المرتبط به.
- المديونية لا تدخل كاش الشفت ولا ربح المتجر افتراضيًا.
- نقل الموظف بين المتاجر لا يغير تقارير الأشهر السابقة.
- PDF الموظف أصبح قراءة فقط ولا يحذف أو يصفر العمليات.

### 2.4 خدمات الواجهات والتنظيف

تم نقل تجهيز بيانات العرض إلى خدمات خفيفة:

- `OwnerDashboardViewService`
- `EmployeeActionsViewService`

وتم تثبيت صفحة عمليات الموظف/المحاسب الحالية على القواعد التالية:

- فلتر الشهر يطبق على كروت الملخص وسجل العمليات وتفاصيل المودال، بينما أزرار تسجيل العمليات نفسها تبقى مستقلة عن الفلتر.
- كل كرت إحصائي يفتح مودال تفاصيل واحدًا بدل صفحات متفرقة.
- زر تصدير بيانات الموظف موجود ضمن كروت العمليات ويعمل كقراءة فقط بدون تصفير.
- تم حذف زر إرسال تقارير للجميع ومساره وكنترولره لأنه كان يحذف بيانات بعد الإرسال.
- تم حذف صفحة السجل الكامل المنفصلة والاكتفاء بسجل أسفل الصفحة مع pagination بعشرة أسطر للشهر المحدد.
- تم حذف كرت بيانات المستخدم من هذه الصفحة، مع إبقاء مكون التفاصيل مستخدمًا فقط في صفحات أعضاء المتجر التي تحتاجه.

وتم حذف/إزالة المسارات القديمة مثل:

- `NotificationQueryService`
- `QuickSaleDeductionCalculator`
- اختبار `QuickSaleDeductionCalculatorTest`
- `AdminReportController`
- صفحة `employees.logs`
- مذكرة لوحة المالك القديمة داخل مجلد الكنترولر

---

## 3. ما لا ينفذ الآن

لا يتم الآن:

1. تغيير صلاحيات المحاسب في السحوبات أو المديونيات.
2. حذف بيانات مالية قديمة من قاعدة البيانات.
3. إزالة fallback القديم من الحسابات قبل التجربة والمقارنة.
4. فتح refactor كبير جديد في `DailySalesController` أو `Accountant\DashboardController` قبل تثبيت النتائج.
5. إضافة migrations جديدة قبل عرض سببها وتأثيرها واعتمادها.

## 4. التجربة اليدوية المطلوبة غدًا

| الواجهة أو العملية | ما يجب تجربته | النتيجة المتوقعة |
|---|---|---|
| لوحة المالك | فتح اللوحة ومقارنة مبيعات/تكلفة/مصروفات اليوم والشهر | الأرقام تظهر من المصدر المالي المشترك بدون أخطاء |
| صفحة عمليات الموظف/المحاسب | فتح الصفحة من المالك والمحاسب حسب الصلاحيات | كروت الملخص والأزرار والسجل تظهر بدون خطأ `operationSummaryCards` |
| فلتر الشهر في صفحة العمليات | تغيير الشهر من أعلى الصفحة | تتغير كروت الملخص وتفاصيل المودالات والسجل فقط، ولا تختفي أزرار تسجيل العمليات |
| مودالات كروت الملخص | الضغط على السحوبات/المديونيات/الآجل/الغياب | تظهر تفاصيل العملية والتاريخ ومن أضافها والملاحظات إن وجدت |
| سجل العمليات أسفل الصفحة | اختيار شهر فيه أكثر من 10 عمليات | يظهر السجل مقسمًا إلى صفحات ويحافظ على الشهر المحدد |
| تصدير بيانات الموظف | الضغط على كرت التصدير ضمن عمليات الموظف | يتم التصدير دون حذف أو تصفير للعمليات السابقة |
| تسجيل سحب | تسجيل عملية ثم محاولة تكرارها | الأولى تحفظ، والتكرار يمنع من الخدمة |
| تسجيل غياب | تسجيل غياب بتاريخ محدد | يظهر في الملخص والتقارير بتاريخ العملية |
| تسجيل مديونية | تسجيل مديونية مكررة بنفس البيانات | تمنع العملية المكررة |
| تحصيل مديونية | تحصيل جزئي ثم كامل | ينقص الرصيد بدون حذف الأصل |
| بيع آجل | إنشاء بيع آجل ثم تحصيله | يتحدث المتبقي والحالة دون حذف السجل |
| PDF الموظف | تصدير التقرير مرتين | لا حذف ولا تصفير بعد التصدير |
| تقرير متجر شهري بعد نقل موظف | موظف في متجر 1 شهر 5 ثم نقل لمتجر 2 شهر 6 | يبقى ظاهرًا في تقرير متجر 1 لشهر 5 |
| لوحة المحاسب/الشفت | فتح آخر العمليات وتفاصيل الشفت | العمليات تظهر حسب `business_date`/الشفت لا `created_at` وحده |

## 5. ملف التنفيذ الذاتي للتنظيف لاحقًا

تمت إضافة أمر Artisan للتنظيف اللاحق:

```bash
php artisan refactor:cleanup-legacy-financial
```

الأمر الافتراضي **dry-run** ولا يحذف شيئًا.

للتنفيذ بعد انتهاء التجربة والموافقة:

```bash
php artisan refactor:cleanup-legacy-financial --apply --confirm=DELETE_LEGACY_FINANCIAL_REFACTOR_ARTIFACTS
```

ما يفعله الأمر:

1. يفحص ملفات/أدوات قديمة معروفة يفترض حذفها بعد الانتقال.
2. يحذف الملفات القديمة المدرجة فقط عند استخدام `--apply` مع كلمة التأكيد.
3. ينشئ تقريرًا في `storage/app/refactor/legacy_financial_cleanup_report.json`.
4. لا يحذف بيانات من قاعدة البيانات.
5. لا يزيل fallback المحاسبي من الخدمات تلقائيًا؛ هذا يحتاج مقارنة أرقام أولًا.

---

## 6. أفكاري للمرحلة التالية بعد التجربة

إذا نجحت التجربة اليدوية، فالمرحلة التالية الأفضل ليست حذفًا عشوائيًا، بل:

1. بناء شاشة/أمر تدقيق يقارن أرقام المصدر القديم والجديد للمتجر والفترة.
2. تحديد هل `SalesCostService` هو الاسم النهائي أم نحتاج `SaleItemCostCalculator` كاسم داخلي لاحق.
3. إزالة fallback القديم من نطاق صغير بعد نجاح المقارنة، وليس من كل المشروع دفعة واحدة.
4. تثبيت اختبارات Feature للمسارات التي ستُعتمد نهائيًا.
5. بعد ذلك فقط يتم تفكيك أي كنترولر ثقيل بقي فيه منطق عرض أو حساب مكرر.

---

## 7. معيار اعتبار هذه المرحلة مغلقة

تعتبر هذه المرحلة مغلقة عندما:

1. تنتهي التجربة اليدوية دون خطأ يمنع الاستخدام.
2. تسجل أي فروقات مالية في قائمة مستقلة.
3. لا توجد تعليمات قديمة متعارضة في المستندات.
4. يتم تشغيل أمر التنظيف لاحقًا فقط بعد الموافقة، وليس قبل التجربة.
